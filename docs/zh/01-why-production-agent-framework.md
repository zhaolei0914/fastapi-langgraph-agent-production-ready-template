# 01 为什么需要生产级 Agent 后端框架

很多人第一次学习 Agent，通常是从一个非常简单的脚本开始：写一个 Prompt，调用一次大模型，如果模型需要工具，就手动执行一个函数，然后把结果再发给模型。这个过程很适合入门，但它和真实业务系统之间有很大的距离。

当前项目的价值就在这里：它不是一个只为了演示概念的 Demo，而是一个面向生产环境的 Agent 后端模板。它把真实上线时必须面对的问题提前纳入架构中，比如会话状态、长期记忆、工具调用、认证、限流、日志、监控、评估和部署。

这一篇先不深入代码，而是帮助你建立一个最重要的认知：为什么 Agent 平台不能只等于“调用大模型 API”。

## 1. 一个最简单的 Agent 是什么

最简单的 Agent 可以被理解成一个循环：

1. 用户提出问题。
2. 系统把问题和 Prompt 发给大模型。
3. 大模型返回回答，或者要求调用工具。
4. 如果要调用工具，程序执行工具。
5. 把工具结果再发给大模型。
6. 大模型生成最终回答。

这个过程看起来不复杂。你甚至可以用几十行 Python 代码实现一个玩具版本。

但是，一旦你想把它给真实用户使用，问题马上就会出现。

## 2. Demo Agent 和生产级 Agent 的差异

### 2.1 Demo Agent 只关心“能不能跑”

一个 Demo 通常只需要满足：

- 能调用模型。
- 能输出回答。
- 能调用一两个工具。
- 能在本地命令行或 Notebook 里演示。

这种 Demo 的目标是验证想法，不是承载真实用户。

### 2.2 生产级 Agent 必须关心“长期稳定运行”

真实生产环境要考虑的问题完全不同：

- **用户是谁**：不同用户的数据和记忆必须隔离。
- **是哪一次会话**：同一个用户可能有多个对话 Session。
- **状态在哪里**：多轮对话不能只放在内存里。
- **模型失败怎么办**：超时、限流、供应商错误都很常见。
- **工具失败怎么办**：工具调用可能慢、可能报错、可能返回脏数据。
- **请求太多怎么办**：需要限流保护系统。
- **输出慢怎么办**：需要流式响应改善体验。
- **出了问题怎么查**：需要日志、Trace、Metrics。
- **回答质量怎么判断**：需要评估体系，而不是凭感觉。
- **如何部署扩容**：不能只靠本地运行。

所以生产级 Agent 后端不是一个函数，而是一套系统。

## 3. 当前项目想解决什么问题

当前项目是一个 FastAPI + LangGraph 的生产级 Agent 后端模板，它的目标是把“每个 Agent 产品都需要反复搭建的基础设施”提前准备好。

项目 README 中对它的定位很明确：

- **LangGraph**：负责有状态、多步骤 Agent 工作流。
- **FastAPI**：负责高性能异步 REST API。
- **PostgreSQL + pgvector**：负责状态存储和长期记忆。
- **mem0**：负责语义记忆管理。
- **Langfuse**：负责 LLM 调用观测。
- **JWT Session**：负责用户认证和会话隔离。
- **Prometheus + Grafana**：负责系统监控。
- **slowapi**：负责接口限流。
- **tenacity**：负责 LLM 调用重试。
- **Docker/Alembic**：负责部署和数据库迁移。

这些能力放在一起，才构成一个接近真实业务的 Agent 后端框架。

## 4. 为什么使用 FastAPI

FastAPI 在这个项目中不是“附属品”，而是整个 Agent 后端的入口。

### 4.1 Agent 需要 API 化

真实产品中，Agent 往往不是自己运行在命令行里，而是被前端、移动端、第三方系统或内部服务调用。调用方式通常是 HTTP API。

FastAPI 负责把 Agent 能力包装成接口，例如：

- 用户注册。
- 用户登录。
- 创建会话。
- 普通聊天。
- 流式聊天。
- 获取历史消息。
- 清空历史消息。

在当前项目中，聊天接口位于：

```text
app/api/v1/chatbot.py
```

认证接口位于：

```text
app/api/v1/auth.py
```

### 4.2 FastAPI 带来的生产能力

FastAPI 的价值不只是路由，还包括：

- 请求参数校验。
- 响应模型约束。
- 依赖注入。
- 自动 OpenAPI 文档。
- 异步处理能力。
- 中间件生态。

对于 Agent 后端来说，这些能力很重要。因为大模型应用本质上也是后端服务，必须符合后端工程的基本要求。

## 5. 为什么使用 LangGraph

如果只调用一次模型，LangChain 或 OpenAI SDK 就够了。那为什么还要 LangGraph？

因为 Agent 不只是一次调用，而是一个有状态的流程。

### 5.1 Agent 有流程

一个 Agent 可能需要：

- 先读取用户输入。
- 查询长期记忆。
- 调用模型判断是否需要工具。
- 执行工具。
- 把工具结果送回模型。
- 必要时再次调用工具。
- 最后生成回答。

这已经不是普通函数调用，而是一个图结构或状态机。

### 5.2 Agent 有状态

Agent 需要知道：

- 当前有哪些消息。
- 当前属于哪个用户。
- 当前属于哪个 Session。
- 是否处于中断等待状态。
- 是否已经调用过工具。
- 长期记忆中有哪些相关内容。

LangGraph 的核心价值就是用 `StateGraph` 把这些状态和流程组织起来。

当前项目的 Agent 图非常简洁：

```text
START -> chat -> tool_call -> chat -> END
```

这意味着：

- `chat` 节点负责调用 LLM。
- 如果 LLM 要调用工具，就进入 `tool_call`。
- 工具结果再回到 `chat`。
- 如果 LLM 不再调用工具，就结束。

这就是一个真实 Agent Loop 的最小生产形态。

## 6. 为什么需要 Checkpoint

很多入门教程会把历史消息放在一个 Python list 里。这样在本地演示可以，但在生产环境不行。

### 6.1 只放内存的问题

如果状态只放在内存中，会有很多问题：

- 服务重启后历史消失。
- 多进程或多实例之间状态不共享。
- 用户刷新页面后可能丢失上下文。
- 无法可靠恢复中断中的 Agent。
- 无法水平扩容。

### 6.2 当前项目的设计

当前项目使用 LangGraph 的 `AsyncPostgresSaver` 把状态保存到 PostgreSQL。

关键文件：

```text
app/core/langgraph/graph.py
```

核心思想是：

- 每个 Session 对应一个 `thread_id`。
- LangGraph 根据 `thread_id` 保存和读取状态。
- PostgreSQL 负责持久化这些状态。

这样，一个会话就不再依赖某个 Python 进程的内存，而是可以被可靠保存和恢复。

## 7. 为什么需要长期记忆

Checkpoint 保存的是会话状态，长期记忆解决的是另一个问题：用户跨会话的偏好和事实。

例如用户之前说过：

- 我喜欢简洁回答。
- 我正在开发一个 FastAPI 项目。
- 我的数据库使用 PostgreSQL。
- 我更喜欢中文解释。

这些信息不应该只存在某一个 Session 中。下次用户新建会话时，Agent 仍然应该能利用这些信息。

当前项目使用：

```text
mem0 + pgvector
```

长期记忆的基本流程是：

1. 对话完成后，后台任务把消息交给 mem0。
2. mem0 抽取有价值的事实。
3. 将事实向量化后存入 PostgreSQL pgvector。
4. 新请求到来时，根据用户问题做语义检索。
5. 把相关记忆注入系统 Prompt。

关键文件：

```text
app/services/memory.py
```

这种设计让 Agent 不只是“当前会话助手”，而更像“了解用户长期上下文的助手”。

## 8. 为什么需要 LLM Service

很多初学者会在业务代码里直接写 OpenAI 调用。但在生产环境中，这样会让系统非常脆弱。

### 8.1 直接调用模型的问题

模型调用可能出现：

- 超时。
- 限流。
- 供应商接口异常。
- 某个模型临时不可用。
- 响应格式不符合预期。
- 工具绑定丢失。

如果每个地方都直接调用模型，就会产生大量重复的错误处理代码。

### 8.2 当前项目的设计

当前项目把模型调用集中到：

```text
app/services/llm/service.py
```

它负责：

- 统一调用模型。
- 使用 tenacity 做重试。
- 支持 circular fallback。
- 控制总超时时间。
- 支持工具绑定。
- 支持结构化输出。

这样，Agent 图不需要关心模型失败后的复杂处理，只需要调用：

```text
llm_service.call(...)
```

这是典型的生产级解耦：Agent 负责流程，LLM Service 负责模型可靠性。

## 9. 为什么需要认证和 Session

如果 Agent 产品面向多个用户，就必须区分：

- 谁在使用。
- 这个请求属于哪个会话。
- 这个用户能不能访问这个 Session。
- 长期记忆属于哪个用户。

当前项目采用两类 token：

- **User token**：用于识别用户，创建和管理 Session。
- **Session token**：用于访问某个具体聊天会话。

这种设计的好处是：

- 用户身份和会话身份分离。
- 每个聊天请求都明确绑定到一个 Session。
- LangGraph 的 `thread_id` 可以直接使用 `session.id`。
- 长期记忆可以通过 `user_id` 隔离。

关键文档：

```text
docs/authentication.md
```

## 10. 为什么需要可观测性

Agent 系统比传统后端更难调试。因为错误可能发生在很多层：

- HTTP 请求层。
- 认证层。
- 数据库层。
- LangGraph 状态层。
- LLM 调用层。
- 工具执行层。
- Prompt 设计层。
- 记忆检索层。

如果没有可观测性，你只能看到“回答不对”或者“接口失败”，但不知道为什么。

当前项目提供了多种观测能力：

- **structlog**：结构化日志。
- **Langfuse**：LLM 调用 Trace。
- **Prometheus**：指标采集。
- **Grafana**：监控面板。
- **ProfilingMiddleware**：慢请求性能分析。

这就是生产级系统和 Demo 的重要差异之一：Demo 追求能跑，生产系统必须可解释、可定位、可恢复。

## 11. 为什么需要评估体系

传统后端可以用单元测试验证函数输出。但 Agent 的输出是概率性的，不同模型、Prompt、上下文都会影响结果。

所以你需要评估体系回答这些问题：

- Agent 有没有帮助用户？
- 是否偏题？
- 是否幻觉？
- 是否有害？
- 是否太啰嗦？
- 工具调用是否合理？
- 改 Prompt 后质量是变好了还是变差了？

当前项目的 `evals/` 目录提供了基于 Langfuse traces 的评估框架。它可以拉取历史调用记录，用 LLM Judge 根据指标进行打分，并生成 JSON 报告。

生产环境中，如果没有评估，你很难持续迭代 Agent。

## 13. 生产级 Agent 框架应该具备哪些能力

基于当前项目，一个生产级 Agent 框架至少应该具备以下能力：

- **API 能力**：通过标准接口对外提供服务。
- **状态能力**：保存多轮对话状态。
- **记忆能力**：跨会话保存用户长期上下文。
- **工具能力**：让模型调用外部函数和业务系统。
- **流式能力**：提升用户等待体验。
- **认证能力**：区分用户、会话和权限。
- **限流能力**：保护系统和模型额度。
- **模型可靠性**：重试、fallback、超时。
- **配置能力**：支持开发、测试、生产多环境。
- **数据库迁移能力**：可控演进数据结构。
- **日志能力**：出现问题能定位。
- **指标能力**：知道系统是否健康。
- **Trace 能力**：知道 LLM 每一步发生了什么。
- **评估能力**：知道 Agent 质量是否提升。
- **部署能力**：可以用 Docker 或云环境运行。

后续 17 篇教程会逐个拆开这些能力。


先读这几个文件：

## 15. 本篇总结

这一篇的核心观点是：生产级 Agent 不是“一个会调用大模型的脚本”，而是一个完整后端系统。

当前项目提供的不是单点能力，而是一套生产级 Agent 后端基础设施：

- FastAPI 负责 API 接入。
- LangGraph 负责 Agent 流程和状态。
- PostgreSQL 负责持久化。
- mem0 + pgvector 负责长期记忆。
- LLM Service 负责模型可靠性。
- JWT Session 负责多用户隔离。
- Langfuse、Prometheus、Grafana 负责可观测性。
- evals 负责质量评估。

如果你想精通 Agent 开发，需要从“Prompt 调参思维”升级为“平台工程思维”。这也是这套教程的主线。
